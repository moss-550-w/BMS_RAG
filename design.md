# BMS领域专业RAG问答系统 纯架构设计方案
本架构严格遵循**学术严谨性、领域适配性、模块化可扩展**三大原则，完全匹配你项目的核心需求（零幻觉、精确溯源、双引擎支持、本地持久化），采用**分层模块化架构**，边界清晰、职责单一，既满足快速交付要求，也预留了未来迭代空间。

---

## 一、整体架构总览
采用**五层垂直分层架构**，从上到下依次为用户交互层、API网关层、应用服务层、RAG核心引擎层、数据持久层。所有层之间通过标准化接口通信，层内组件高内聚、层间低耦合。

```
┌─────────────────────────────────────────────────────────┐
│                    用户交互层 (UI Layer)                 │
│  三栏式界面 | 问答展示 | 配置面板 | 溯源面板 | 状态展示  │
└───────────────────────────┬─────────────────────────────┘
                            ↓ HTTP/REST
┌─────────────────────────────────────────────────────────┐
│                    API网关层 (Gateway Layer)             │
│  请求路由 | 参数校验 | 错误统一处理 | 跨域 | 流量控制    │
└───────────────────────────┬─────────────────────────────┘
                            ↓ 业务编排
┌─────────────────────────────────────────────────────────┐
│                   应用服务层 (Service Layer)             │
│  问答服务 | 索引管理服务 | 配置管理服务 | 状态服务       │
└───────────────────────────┬─────────────────────────────┘
                            ↓ 核心能力调用
┌─────────────────────────────────────────────────────────┐
│                  RAG核心引擎层 (RAG Engine Layer)        │
│  数据预处理子系统 | 检索子系统 | 生成子系统 | 领域增强    │
└───────────────────────────┬─────────────────────────────┘
                            ↓ 数据读写
┌─────────────────────────────────────────────────────────┐
│                   数据持久层 (Storage Layer)             │
│  FAISS向量库 | 论文元数据库 | 本地文件系统 | 配置存储     │
└─────────────────────────────────────────────────────────┘
```

## 二、各层详细架构设计
### 1. 用户交互层
**核心职责**：仅负责用户输入接收与结果可视化展示，不包含任何业务逻辑。
- **界面组件**：
  - 顶部导航组件：系统标题、全局状态指示器、全局操作按钮（清空对话、重建索引、主题切换）
  - 左侧配置组件：向量引擎选择器、检索参数调节器、文献资源列表、系统说明
  - 中间问答组件：对话历史展示区、流式回答渲染器、专业术语高亮器、引用角标渲染器
  - 右侧溯源组件：当前回答引用列表、原文片段预览器、全局文献目录
- **交互规范**：所有用户操作均通过标准化API请求发送至网关层，不直接调用任何后端逻辑；所有后端返回数据均通过统一渲染器展示。

### 2. API网关层
**核心职责**：统一入口，隔离前端与后端业务逻辑，提供基础网关能力。
- **核心组件**：
  - 请求路由器：将不同URL请求分发至对应应用服务
  - 参数校验器：校验所有入参的合法性（如Top-K范围、引擎名称枚举）
  - 错误统一处理器：捕获所有后端异常，转换为标准化错误码返回前端
  - 跨域资源共享(CORS)处理器：解决前后端分离部署的跨域问题
- **接口定义**：仅暴露3类核心接口，严格遵循RESTful规范：
  1. 问答类接口：`POST /api/v1/query`（支持流式与非流式两种模式）
  2. 管理类接口：`POST /api/v1/index/rebuild`、`POST /api/v1/index/update`
  3. 状态类接口：`GET /api/v1/status`

### 3. 应用服务层
**核心职责**：业务编排层，负责组合RAG核心引擎的能力，实现具体业务流程，不包含任何RAG底层逻辑。
- **核心服务**：
  - **问答服务**：接收用户问题，调用检索子系统和生成子系统，组装答案与引用数据返回
  - **索引管理服务**：负责向量索引的重建、增量更新、加载与卸载
  - **配置管理服务**：管理全局配置（如API密钥、默认参数），支持配置热更新
  - **系统状态服务**：收集并返回向量库状态、模型运行状态、论文数量等系统信息
- **设计原则**：每个服务对应一个独立的业务场景，服务之间无直接依赖，仅通过RAG核心引擎交互。

### 4. RAG核心引擎层
**项目最核心层**，所有BMS领域定制化能力均在此层实现，进一步拆分为4个独立子系统。

#### 4.1 数据预处理子系统
**核心职责**：将原始PDF论文转换为可检索的标准化文本块。
- **组件**：
  - PDF解析器：提取PDF文本内容与页码信息
  - BMS专属清洗器：处理分页打断、特殊符号、算法公式，保留专业术语完整性
  - 文本分块器：固定分块大小（250字符，适配API限制），支持重叠分块
  - 元数据生成器：为每个文本块附加论文编号、页码、章节信息

#### 4.2 检索子系统
**核心职责**：精准召回与用户问题相关的论文片段。
- **组件**：
  - 多引擎嵌入适配器：统一封装火山引擎Ark与阿里云DashScope嵌入模型，支持动态切换
  - FAISS检索器：执行向量相似度检索，返回Top-K候选片段
  - Rerank重排器：对候选片段进行二次排序，提升相关性
  - 检索结果处理器：过滤低相关性片段，组装元数据返回

#### 4.3 生成子系统
**核心职责**：基于检索结果生成零幻觉、结构化的学术回答。
- **组件**：
  - 大模型适配器：统一封装火山引擎Doubao-32K大模型接口
  - 专家级Prompt管理器：管理BMS领域专属System Prompt，严格限制模型仅使用上下文回答
  - 结构化回答生成器：将模型输出转换为"核心结论+详细阐述+引用列表"的标准化格式
  - 引用注入器：将引用标记自动插入到回答对应的技术观点处

#### 4.4 BMS领域增强模块
**项目差异化核心**，独立于通用RAG逻辑之外。
- **组件**：
  - BMS查询重写器：将口语化提问转换为学术检索式，优化检索效果
  - 专业术语词典：维护BMS领域术语表，用于术语高亮与检索优化
  - 引用溯源管理器：统一管理所有引用信息，确保引用标记与溯源面板的一致性

### 5. 数据持久层
**核心职责**：负责所有数据的本地持久化存储，不依赖任何外部数据库。
- **存储组件**：
  - FAISS向量库：存储文本块的向量表示，支持本地保存与加载
  - 论文元数据库：存储所有论文的基础信息（标题、作者、页码范围）与文本块元数据
  - 本地文件系统：存储原始PDF文件、FAISS索引文件、配置文件
  - 环境变量配置存储：存储API密钥、接入点ID等敏感配置

## 三、核心业务流程架构
### 1. 标准问答全流程
```
用户输入问题 → 前端校验 → 发送POST /api/v1/query
    ↓
API网关层：参数校验 → 路由至问答服务
    ↓
应用服务层：问答服务接收请求
    ↓
RAG核心引擎层：
    1. BMS查询重写器重写问题
    2. 多引擎嵌入适配器生成问题向量
    3. FAISS检索器召回Top-K片段
    4. Rerank重排器二次排序
    5. 大模型适配器基于上下文生成回答
    6. 结构化回答生成器组装答案与引用
    ↓
应用服务层：返回标准化响应
    ↓
API网关层：统一错误处理 → 返回前端
    ↓
前端：渲染回答 → 高亮引用标记 → 右侧展示溯源面板
```

### 2. 向量索引重建流程
```
用户点击"重建索引" → 前端确认 → 发送POST /api/v1/index/rebuild
    ↓
API网关层：路由至索引管理服务
    ↓
应用服务层：索引管理服务启动重建任务
    ↓
RAG核心引擎层：
    1. PDF解析器批量解析所有PDF
    2. BMS专属清洗器清洗文本
    3. 文本分块器生成标准化文本块
    4. 多引擎嵌入适配器批量生成向量
    5. FAISS检索器构建向量索引
    ↓
数据持久层：保存FAISS索引文件与元数据
    ↓
应用服务层：更新系统状态
    ↓
前端：展示重建完成提示 → 刷新系统状态
```

### 3. 多引擎动态切换流程
```
用户在前端选择嵌入引擎 → 前端保存配置
    ↓
下次发起问答请求时，将引擎参数带入请求体
    ↓
API网关层：校验引擎参数合法性
    ↓
应用服务层：问答服务将引擎参数传递给检索子系统
    ↓
RAG核心引擎层：多引擎嵌入适配器根据参数切换对应模型
    ↓
使用选定引擎执行向量检索与生成
```

## 四、非功能性架构设计
### 1. 性能优化架构
- **批量向量化优化**：索引重建时采用批量向量化，减少API调用次数
- **FAISS索引优化**：采用IVF_FLAT索引，平衡检索速度与精度
- **缓存机制**：缓存常用问题的检索结果，提升重复提问的响应速度
- **流式输出优化**：采用SSE（服务器发送事件）实现流式回答，降低用户等待感知

### 2. 可靠性架构
- **API重试机制**：对云服务商API调用实现指数退避重试
- **错误降级**：当Rerank服务不可用时，自动降级为仅使用向量检索
- **数据备份**：自动备份FAISS索引文件与元数据，支持快速恢复
- **状态监控**：实时监控模型API可用性与向量库状态

### 3. 可扩展性架构
- **插件化引擎设计**：新增嵌入模型或大模型时，仅需实现对应适配器接口
- **模块化组件**：所有组件均为独立模块，可单独替换或升级
- **增量更新支持**：支持新增论文时仅向量化新内容，无需重建整个索引
- **接口版本化**：所有API均带版本号，支持平滑升级

### 4. 安全性架构
- **API密钥隔离**：所有敏感配置均存储在环境变量中，不硬编码在代码里
- **输入校验**：严格校验所有用户输入，防止注入攻击
- **数据隔离**：本地数据仅存储在项目目录下，不对外暴露
- **访问控制**：单机部署时默认仅本地访问，生产环境可添加身份认证

## 五、部署架构设计
### 1. 单机一体化部署（推荐，适配你的项目）
所有组件部署在同一台机器上，适合开发、演示与单用户使用。
```
┌─────────────────────────────────────────┐
│              单机服务器                  │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ Streamlit│  │ FastAPI │  │ FAISS   │  │
│  │ 前端     │←→│ 后端    │←→│ 向量库  │  │
│  └─────────┘  └─────────┘  └─────────┘  │
│               ↑ ↓                       │
│          本地文件系统                    │
│  (原始PDF、索引文件、配置文件)           │
└─────────────────────────────────────────┘
```

### 2. 前后端分离部署（生产环境）
前端与后端分离部署，支持多用户并发访问。
```
┌─────────────┐    HTTP    ┌─────────────┐
│ 前端服务器  │ ←────────→ │ 后端服务器  │
│ (Streamlit) │            │ (FastAPI)   │
└─────────────┘            └──────┬──────┘
                                 ↓
                        ┌─────────────┐
                        │ 存储服务器  │
                        │ (FAISS+文件)│
                        └─────────────┘
```

## 六、架构核心优势
1. **零幻觉保证**：生成层严格限制模型仅使用检索到的上下文回答，Prompt中明确禁止引入外部知识
2. **精确溯源能力**：从数据预处理到生成全链路保留元数据，确保每个技术观点都能追溯到具体论文页码
3. **多引擎无缝切换**：通过适配器模式统一不同云服务商接口，配置一行即可切换
4. **快速开发与维护**：分层架构职责清晰，代码复用率高，问题定位与修复简单
5. **BMS领域深度适配**：独立的领域增强模块，针对性解决BMS论文解析与检索的痛点
